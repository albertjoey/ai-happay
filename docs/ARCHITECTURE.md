## 微服务架构

### 用户服务 (User Service)

**职责**:
- 用户注册、登录、认证
- 用户信息管理
- 关注关系管理
- 第三方登录集成

**技术栈**:
- Go + go-zero
- MySQL (用户数据)
- Redis (Session、Token缓存)
- JWT (认证)

**API端口**: 8001

**核心功能**:
```
用户服务
├── 认证模块
│   ├── 注册
│   ├── 登录
│   ├── 第三方登录
│   └── Token刷新
├── 用户管理
│   ├── 用户信息查询
│   ├── 用户信息更新
│   └── 用户列表
└── 关系管理
    ├── 关注用户
    ├── 取消关注
    ├── 关注列表
    └── 粉丝列表
```

### 内容服务 (Content Service)

**职责**:
- 内容发布、编辑、删除
- 内容列表查询
- 媒体资源管理
- 话题、标签管理

**技术栈**:
- Go + go-zero
- MySQL (内容数据)
- Redis (热门内容缓存)
- Kafka (内容事件)

**API端口**: 8002

**核心功能**:
```
内容服务
├── 内容管理
│   ├── 创建内容
│   ├── 更新内容
│   ├── 删除内容
│   ├── 内容详情
│   └── 内容列表
├── 媒体管理
│   ├── 上传媒体
│   ├── 删除媒体
│   └── 媒体列表
├── 话题管理
│   ├── 创建话题
│   ├── 话题列表
│   └── 话题内容
└── 标签管理
    ├── 创建标签
    ├── 标签列表
    └── 标签内容
```

**核心功能**:
```
互动服务
├── 点赞管理
│   ├── 点赞
│   ├── 取消点赞
│   └── 点赞列表
├── 收藏管理
│   ├── 收藏
│   ├── 取消收藏
│   └── 收藏列表
├── 评论管理
│   ├── 发表评论
│   ├── 删除评论
│   └── 评论列表
├── 消息通知
│   ├── 消息列表
│   ├── 标记已读
│   └── 未读数量
└── 观看历史
```

### 数据分片策略

**租户隔离**:
- 每个租户数据通过 `tenant_id` 字段隔离
- 支持按租户分库分表

**内容分表**:
- 内容表按时间分表（按月）
- 互动表按用户ID分表

## 缓存策略

### Redis缓存设计

```
Redis
├── 用户缓存
│   ├── user:info:{user_id}          # 用户信息
│   ├── user:token:{user_id}         # 用户Token
│   └── user:stats:{user_id}         # 用户统计
├── 内容缓存
│   ├── content:info:{content_id}    # 内容详情
│   ├── content:hot                  # 热门内容列表
│   └── content:recommend:{user_id}  # 推荐内容
├── 计数器
│   ├── count:like:{content_id}      # 点赞数
│   ├── count:view:{content_id}      # 浏览数
│   └── count:comment:{content_id}   # 评论数
└── Session
    └── session:{session_id}         # 会话信息
```

### 缓存更新策略

1. **Cache-Aside**: 读时更新，写时失效
2. **Write-Through**: 写时同步更新缓存
3. **Write-Behind**: 异步更新缓存

## 消息队列设计

### Kafka Topic设计

```
Kafka Topics
├── content.created      # 内容创建事件
├── content.updated      # 内容更新事件
├── content.deleted      # 内容删除事件
├── interaction.like     # 点赞事件
├── interaction.comment  # 评论事件
├── user.follow          # 关注事件
└── notification.send    # 消息通知事件
```

### 事件处理流程

```
用户点赞 → 互动服务 → Kafka → 消息服务 → 推送通知
                      ↓
                   搜索服务 → 更新索引
                      ↓
                   统计服务 → 更新统计
```

## 安全设计

### 认证授权

```
认证流程:
1. 用户登录 → 验证账号密码
2. 生成JWT Token → 返回给客户端
3. 客户端请求 → 携带Token
4. 网关验证Token → 转发请求
5. 服务处理请求 → 返回响应
```

### 权限控制

- RBAC (基于角色的访问控制)
- 资源级权限控制
- 操作级权限控制

### 数据安全

- 敏感数据加密存储
- HTTPS传输加密
- SQL注入防护
- XSS防护
- CSRF防护

### 告警策略

- 错误率 > 1%: 警告
- 错误率 > 5%: 严重
- 响应时间 > 1s: 警告
- 响应时间 > 3s: 严重
- CPU > 80%: 警告
- CPU > 95%: 严重

## 总结

Happy系统采用微服务架构，具有以下特点:

1. **高可用**: 服务多副本部署，自动故障转移
2. **高性能**: 多级缓存，异步处理，水平扩展
3. **高扩展**: 微服务架构，易于扩展新功能
4. **高安全**: 多层安全防护，数据加密
5. **易维护**: 服务拆分，职责清晰，文档完善
